# ADR-0006: librustzcash Ironwood/NU6.3 Dependency Family

| Field | Value |
|-------|-------|
| Status | Accepted on 2026-07-04 |
| Product | Zally |
| Domain | librustzcash dependency versioning, patch-source topology, NU6.3 network plumbing |
| Related | [ADR-0001 Workspace crate boundaries](0001-workspace-crate-boundaries.md); [ADR-0001 Zinder-only ChainSource](0001-zinder-only-chain-source.md); [Public interfaces](../architecture/public-interfaces.md) |

## Context

zinder v0.2.0 (the only `ChainSource` backend zally ships, per ADR-0001) moved its own `zebra-chain` dependency onto an unreleased librustzcash family that carries NU6.3/Ironwood support: a new shielded pool and the v6 transaction format (ZIP-230), landed ahead of ZIP-230's own drafted NU7/OrchardZSA scope. Once zally tracked that zinder revision, `zebra-chain` pulled `orchard 0.15.0-pre.2` and `zcash_primitives 0.29.0-pre.0` transitively into the workspace, alongside zally's own direct pins on the older `0.14`/`0.28`-series librustzcash crates. Two incompatible families of the same crates in one dependency graph is not a state cargo can resolve; zally's own `zcash_*` pins had to move onto the same unreleased family zinder already adopted.

None of `zcash_client_backend`, `zcash_client_sqlite`, or `pczt` has a released version (stable or prerelease) against this family; their next release is CHANGELOG-marked "PLANNED" and exists only on librustzcash's unreleased `main`. zally previously carried a single-crate patch on `zcash_client_backend` 0.23.0 (a `target_expiry_height` parameter on `create_pczt_from_proposal`, packaged from a forked commit and consumed via `[patch.crates-io]`) built so that its transitive dependencies resolved from crates.io rather than from a full librustzcash workspace checkout. Upstream librustzcash now carries that target-expiry API plus Ironwood wallet scanning and storage support, so the stopgap fork is no longer the source of truth.

That packaging choice does not survive the family bump. `zcash_client_sqlite` and `pczt` are librustzcash workspace members that resolve `zcash_client_backend` (and every other intra-workspace crate: `zcash_protocol`, `zcash_primitives`, `zcash_keys`, `zcash_address`, `zcash_transparent`, `zcash_proofs`, `zip321`) via `path = "../<crate>"` inside their own git checkout, not via crates.io. Patching `zcash_client_backend` from one git source (the single-crate export) while patching `zcash_client_sqlite`/`pczt` from a *different* git source (a full librustzcash checkout) gives each side a distinct, path-resolved `zcash_client_backend` type. `WalletDb` (typed against the checkout's own `zcash_client_backend`) would then fail to satisfy `WalletWrite`/`InputSource` bounds written against the export's `zcash_client_backend`, the same class of failure this workspace already guards against at the zinder/wallet boundary, reintroduced inside the patch set itself.

## Decision

1. **Every librustzcash-internal crate zally references, directly or transitively, is patched from one git commit.** `[patch.crates-io]` in the root `Cargo.toml` points `zcash_client_backend`, `zcash_client_sqlite`, `pczt`, `zcash_protocol`, `zcash_primitives`, `zcash_keys`, `zcash_address`, `zcash_transparent`, `zcash_proofs`, `zip321`, `zcash_encoding`, and `equihash` at upstream `https://github.com/zcash/librustzcash.git` commit `8e6864a3c67cab3c64a052dd20f83c553662e8b2`. There is no per-crate export step: Cargo resolves each as a workspace member of the one checkout, so every one of them, and everything they depend on internally, comes from a single source. `zcash_encoding` and `equihash` are not zally-direct dependencies; they are patched because they are workspace members of the same checkout and would otherwise resolve from crates.io on an unpatched path, producing the duplicate sources Decision 2 forbids. The prior single-crate `zcash_client_backend-target-expiry` export is retired.

2. **Duplicate-source resolution is the enforced invariant, not a one-time check.** The output of `cargo tree -d -e no-dev --all-features`, filtered to the `zcash_*`/`pczt`/`orchard`/`sapling-crypto`/`equihash` family, must show zero duplicated crates. There is no standing exception within the family: all twelve crates in Decision 1 patch from the same commit, so none of them can appear twice. Adding a new librustzcash-internal dependency, or dropping the last consumer of an existing one, requires re-running this check before merging; any family crate appearing twice blocks the change. The unfiltered command also reports the bip32 pre-release cluster described in Decision 5; that cluster sits outside the family and outside this gate.

3. **Version requirements track the patched commit's actual `package.version`.** Crates whose `Cargo.toml` at the pinned commit still declares an old-family version (`zcash_client_backend = "0.23"`, `zcash_client_sqlite = "0.21"`, `pczt = "0.7"`, `zip321 = "0.8"`) keep their existing workspace requirement; the patch only redirects the source, not the version match. Crates whose `Cargo.toml` at the pinned commit declares a new prerelease (`zcash_keys = "0.15.0-pre.0"`, `zcash_primitives = "0.29.0-pre.0"`, `zcash_protocol = "0.10.0-pre.0"`, `zcash_address = "0.13.0-pre.0"`, `zcash_transparent = "0.9.0-pre.0"`, `zcash_proofs = "0.29.0-pre.0"`) have their workspace requirement bumped to match. `orchard` is bumped to `0.15.0-pre.2` and patched to upstream `https://github.com/zcash/orchard` commit `475ef0ff77d45aebff93cb039d639250d82518a3` because the pinned librustzcash commit requires the bundle-type APIs that have not shipped on crates.io yet. `secp256k1`, `sapling-crypto`, `zip32`, `shardtree`, `incrementalmerkletree`, `zcash_script` are unaffected: their pinned ranges already satisfy what the new family requests.

4. **Mainnet stays out of scope until upstream ships a stable release.** Regtest leaves NU6.3 unactivated by zally's own choice (`nu6_3: None`), and mainnet has no activation height defined upstream yet. Testnet is different: upstream's `TestNetwork` hardcodes NU6.3 activation at height 4,134,000, and zally's `Network::Testnet` delegates directly to it with no override. Once the public testnet chain reaches that height, the proposal-execution path shared by `create_proposed_transactions` and `create_pczt_from_proposal` routes any Orchard-recipient proposal through the new Ironwood bundle. `create_pczt_from_proposal` now builds a V6 PCZT and threads Ironwood spends and outputs through `Updater::update_ironwood_with`, and `zally-pczt`'s Signer and Prover apply Ironwood spend authorizations and proofs alongside Sapling and Orchard, selecting the Orchard proving key's circuit version from the PCZT's own consensus branch rather than assuming a fixed one. This closes what was previously an accepted availability gap (`create_pczt_from_proposal` rejecting any Ironwood-touching proposal with `Error::ProposalNotSupported`); a live regtest round trip covering funding, shielding, a standard shielded send, and a PCZT-signed send against an NU6.3-active chain verifies the full path. Enabling NU6.3-active mainnet operation is a separate, explicit decision this ADR does not make.

5. **A second dependency split exists outside this gate's scope, and stays that way pending upstream.** `zcash_transparent`'s `Cargo.toml` pins `bip32 = "=0.6.0-pre.1"` unchanged from upstream librustzcash. That pin carries its own exact-pinned pre-release chain: `sha2 0.11.0-pre.4`, `hmac 0.13.0-pre.4`, `ripemd 0.2.0-pre.4`, `digest 0.11.0-pre.9`, `block-buffer 0.11.0-rc.3`, `crypto-common 0.2.0-rc.1`. The rest of the workspace resolves the now-stable `sha2`/`hmac`/`ripemd`/`digest` lines, so `cargo tree -d` reports six duplicated crates in this cluster on every run. None of the six crosses a boundary zally's own code touches: `bip32` is reachable only through `zcash_transparent`'s `transparent-inputs` feature, and zally never imports `sha2`, `hmac`, `ripemd`, `digest`, `block-buffer`, or `crypto-common` directly. The family gate in Decision 2 is scoped to `zcash_*`/`pczt`/`orchard`/`sapling-crypto`/`equihash` and does not cover this cluster; extending it would mean patching a crate (`bip32`) zally does not reference, for a duplication with no live consequence. The lever that resolves it sits upstream: `bip32`'s home (`iqlusioninc/crates`) has not published a release against the now-stable RustCrypto 0.11/0.13 line, so `zcash_transparent`, and this pinned line with it, has nothing compatible to repoint to. Once that release exists, bumping the workspace `bip32` requirement and dropping the `block-buffer`/`crypto-common` exact pins closes the cluster in one move.

6. **Downstream consumers move in lockstep.** Any consumer building on top of a zally commit that carries this ADR (notably fauzec) must bump its own `zcash_*`/`pczt` pins and its own copy of the `target_expiry_height` patch to the same librustzcash commit zally pins. Picking up zally's API changes without the matching dependency family produces the same two-source-identity failure this ADR exists to avoid.

## Consequences

- zally now tracks an unreleased, prerelease librustzcash family. `cargo update` inside this family can pick up new prerelease point releases (e.g. a future `orchard 0.15.0-pre.3`) without any zally action; a move to a *different* librustzcash commit (picking up further upstream changes) requires re-verifying Gate 2 and re-running the full validation gate, but does not require touching the patch topology itself.
- `zally-storage`'s `WalletStorage::put_subtree_roots` and the `ext_zally_holds.locked_notes` wire format both type-check against a three-variant `ShieldedPool` (`Sapling`, `Orchard`, `Ironwood`) instead of two. Subtree-root priming now records Sapling, Orchard, and Ironwood roots through the corresponding upstream write paths. The locked-notes wire format reserves tag `2` for Ironwood so reservation metadata stays stable as live Ironwood notes appear.
- Dropping the target-expiry export means one fewer git remote to keep in sync, at the cost of a heavier git checkout per patched crate (the full librustzcash workspace, not a single packaged crate) for anyone building zally from source.
