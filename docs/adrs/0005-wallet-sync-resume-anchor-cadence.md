## Context

`Wallet::sync_inner` resumes scanning from the wallet's stored `fully_scanned_height`. The `zcash_client_backend::data_api::wallet::get_wallet_summary` API computes spendable balance using the wallet's stored chain tip and a confirmations policy: per ZIP-315 the trusted-output anchor sits at `chain_tip + 1 - 3`. The sqlite backend additionally refuses to report any shielded balance as spendable if the shard containing the anchor has unscanned ranges at or below the anchor height. This safety check is correct: a wallet that has not scanned past its own anchor cannot rule out double-spends in the unscanned range.

The first revision of this ADR introduced a two-tip contract: `chain_tip()` returned the live head, `safe_chain_tip()` returned the head minus the reorg window (~100 blocks). The wallet was told to scan only to `safe_chain_tip` so that scanned data could never reorg. Anchor selection used the same tip, and tx-expiry math used the unfiltered head via a separate `update_chain_tip(chain_tip)` call.

In production at testnet height 4033207 (2026-05-27) the wallet held 6,764 TAZ of unspent change outputs from the auto-shield loop, the `inflow_event_recorded` stream confirmed every note was observed in scanned blocks, and `Inflow.Summary.confirmed_balance_zat` (computed from `list_unspent_shielded_notes`) reported the full balance. The dispense path returned `faucet_dry` on every request. `/diagnostics` showed `spendable_zat: 0`. The cause was the anchor-versus-scan-ceiling gap: with `update_chain_tip(real_chain_tip = 4033207)` the anchor sat at 4033205 (trusted=3), the scan ceiling sat at `safe_chain_tip = 4033107`, and librustzcash zeroed the spendable view because the 93 blocks below the anchor were unscanned. Pinning `update_chain_tip` to `safe_chain_tip` instead would have closed that gap but reopened the `BadExpiryHeight` class: tx-expiry sits at `chain_tip + tx_expiry_delta` and would have landed ~80 blocks below the live head, so every shielding and dispense submission would have been rejected by consensus.

The architectural conflict is intrinsic to the librustzcash API: `WalletDb::update_chain_tip` writes a single height that drives both the anchor (which requires the scan to have reached or passed it) and the tx-expiry target (which requires the chain's live head). The two-tip split could not resolve this without exposing the wallet to one or the other failure class.

## Decisions

1. **The wallet scans to the live chain tip.** `Wallet::sync_inner` calls only `chain.chain_tip()`. The result drives `record_observed_tip`, `update_chain_tip`, and the compact-block fetch ceiling. Scanning reaches the head; the wallet's `fully_scanned_height` advances within ~one block of `chain_tip` in steady state; anchors land in scanned territory; `spendable_value()` returns the real balance.

2. **Reorg recovery is automatic within the librustzcash rewind cap (100 blocks = `COINBASE_MATURITY`).** `WalletStorage::scan_blocks` continues to surface `StorageError::ChainReorgDetected { at_height }` when `zcash_client_backend::scan_cached_blocks` reports a continuity error. Posture is now `Retryable`. The sync loop catches the variant, calls `truncate_to_height(at_height - 1)`, emits `WalletEvent::ReorgDetected`, and returns an outcome whose `reorgs_observed > 0` triggers another sync tick. The next tick re-scans from the rewound height. Reorgs deeper than the rewind cap (rare on testnet, ~never on mainnet) surface from `truncate_to_height` as `NotRetryable` and require operator action.

3. **`ChainSource::safe_chain_tip()` and zinder's `LatestSafeBlock` RPC remain on the trait surface but are no longer used by sync.** They serve downstream consumers that need a reorg-robust confirmation signal independent of the wallet's scan progress: fauzec's dispense reaper uses `safe_chain_tip` to decide when a broadcasted transaction has definitively expired (a one-block reorg near the head must not flip a `failed` row back to `submitted`). This is the same head/safe split Ethereum's Engine API exposes. The semantics remain "highest height past the reorg window," produced by zinder against `reorg_window_blocks`.

4. **`update_chain_tip` uses the live head.** Tx-construction math (target height, expiry delta) reads the same `chain_tip`. There is no separate "expiry-math" path; the conflict that motivated the first revision's split is now closed because the scan ceiling is the same value.

5. **`fauzec::WalletPlane::safe_chain_tip_height` reads from `ChainSource::safe_chain_tip()` directly, not from `WalletStatus::safe_chain_tip_height`.** The wallet status field now reports the scan ceiling (= live head), so a consumer that wants the reorg-robust tip must go to the chain source. The dispense reaper's `stalled_broadcast_failure_reason` continues to receive the safe tip and grants `tx_expiry_height + BROADCAST_EXPIRY_GRACE_BLOCKS` of head room before declaring a claim expired.

## Consequences

- The `spendable_zat: 0` wedge observed on 2026-05-27 is closed at its root: the wallet scans to the head, the anchor lands in scanned territory, and `get_wallet_summary(ConfirmationsPolicy::default()).spendable_value()` reports the real balance.
- The `BadExpiryHeight` class is closed by the same change: `update_chain_tip(chain_tip)` is the live head, so the proposal builder's target height is `head + 1` and expiry sits ~40 blocks above the head, well within consensus.
- The faucet's spendable balance now follows the chain in real time (~75-second testnet block cadence). Auto-shield outputs become spendable at ZIP-315's trusted depth (3 confirmations).
- Reorgs that the chain source occasionally serves above the wallet's last-scanned height are auto-recoverable: the wallet rewinds and re-scans. Operators see `WalletEvent::ReorgDetected` for each occurrence. Deep reorgs (>100 blocks) remain operator-visible; resetting `wallet.db` (preserving sealed seed material) is the recovery path.
- `safe_chain_tip` stays first-class for consumers that need it. Splitting head from safe-tip in the dispense reaper means a one-block reorg cannot flip a freshly-expired claim row back into the in-flight state, and a future "deeply-confirmed" UI affordance (e.g., showing receipts only after the tx has aged past the reorg window) can subscribe to the same signal without coupling to wallet sync progress.
- The vocabulary alignment from the first revision (`safe_tip_height` in zinder, `safe_chain_tip_height` in zally's status payload) is unchanged on the wire, but the wallet-status field's semantics shifted: it now reports the scan ceiling, which equals the live chain head, not the safe tip. A subsequent rename pass (`WalletStatus::safe_chain_tip_height` -> `chain_tip_height`, `ChainEvent::SafeChainTipAdvanced` -> `ChainTipAdvanced`) is deferred but not blocking; downstream consumers that want the reorg-robust tip already go through `ChainSource::safe_chain_tip()`.

## Operator runbook

`StorageError::ChainReorgDetected` is now `Retryable` and recovered in-loop; it should not surface to operators except in metrics. If the wallet wedges on a `truncate_to_height` failure (deeper-than-cap reorg) the operator sees `wallet_sync_snapshot_error` with `error="... truncate_to_height failed ..."` and posture `NotRetryable`:

1. Confirm the chain source is healthy. Query `LatestBlock` and `LatestSafeBlock` from zinder; both should advance steadily.
2. Reset `wallet.db` while preserving sealed key material (`wallet.age`, `wallet.age.age-identity`). The wallet will re-scan from birthday.
3. The circuit breaker is not involved: `NotRetryable` postures are excluded from breaker counting. Other wallet operations remain available; only sync is paused.

## Alternatives considered

- **Keep scanning to `safe_chain_tip` and rewrite tx-expiry math outside librustzcash.** Rejected: requires rebuilding signed transactions to swap their `expiry_height`, which invalidates the signature. The wallet would also have to re-fee and re-prove for every expiry adjustment, which is impractical.
- **Use `ConfirmationsPolicy::MIN` (trusted=1, untrusted=1).** Reduces the anchor depth but does not change the underlying requirement that `fully_scanned >= anchor_height`. Did not address the wedge.
- **Lower zinder's `reorg_window_blocks` so `safe_chain_tip` sits close to the head.** Defers but does not remove the conflict: any non-zero safe-tip lag re-creates the anchor gap proportionally.
- **Keep the safe-chain-tip scan contract and accept `spendable_zat = 0` in steady state, expecting operators to pre-fund inventory and tolerate the lag.** Rejected: the faucet's auto-shield loop is the only mechanism that makes the wallet spendable, and it cannot front-fund inventory because it produces only one auto-shield per interval (currently 600s). A faucet whose advertised inventory is permanently zero is a regression versus the original (pre-ADR) behavior.
- **Petition librustzcash to expose a `target_height` parameter on `get_wallet_summary`.** The right long-term move; deferred. The current decision unblocks the user-visible failure without preventing the upstream improvement later.
