## Context

`Wallet::sync_inner` resumes scanning from the wallet's stored `fully_scanned_height`. `zcash_client_backend::data_api::chain::scan_cached_blocks` requires the cumulative `ChainState` at `scanned_from - 1` as input and panics with `assert_eq!(from_height, from_state.block_height() + 1)` when the supplied state does not match the requested resume point.

`zcash_client_backend-0.22.0` does not expose a public API for retrieving a `ChainState` from the wallet's own storage at an arbitrary previously-scanned height. The wallet therefore depends structurally on the configured `ChainSource::tree_state_at` for every sync resume.

The original chain-source contract had a hidden mismatch. The wallet asked `chain_tip()` for its scan ceiling and `tree_state_at(h)` for the anchor at the boundary. Both methods looked symmetric. They were not. In zinder, `chain_tip()` returned the unfiltered visible tip (`ChainEpoch::tip_height`, current as of the latest commit), while `tree_state_at(h)` silently filtered to commitments whose underlying block had aged past the configured reorg window (~100 blocks). That filter is the right design on the producer side: tree-state checkpoints whose block can still reorg out should not be served to a wallet that would anchor on them. But the asymmetry was unspoken; nothing in the trait said the wallet must not scan into the unfinalized window.

The result was a wedge class. The wallet would scan compact blocks all the way to `chain_tip()`, advance `fully_scanned_height` past the reorg-window boundary, then ask `tree_state_at(scanned_from - 1)` on the next pass. The chain source returned its highest visible checkpoint, which was up to `reorg_window_blocks` below the request. The wallet then called `WalletDb::truncate_to_height` against that lower height, the librustzcash 100-block rewind cap rejected `NotRetryable`, and `sync_inner` returned early on every subsequent tick. The wallet's `fully_scanned_height` never advanced, the funding plane's spendable-balance snapshot stayed stale, and the operator-facing surface returned `wallet_insufficient_funds` while the chain source itself was perfectly healthy. fauzec hit this on 2026-05-26 at 04:18 UTC at testnet height 4028669; recovery required a manual wallet-DB reset.

Three earlier attempts targeted the symptom rather than the contract: a consumer-side anchor cache, a denser checkpoint stride, and a defense-in-depth realign helper that truncated to whatever the chain source returned. The first was structurally inert (every successful sync moves `prior_height` forward, so the cache row never matches the next request). The second made more checkpoints exist but did not make them visible. The third absorbed the producer's contract violation by silently rewinding the wallet, which masked the bug.

## Decision

1. **The chain-source surface exposes one tip-like concept, named `safe_chain_tip`.** `ChainSource::safe_chain_tip()` returns the highest height `T` such that, for every `h` in `[T - REWIND_CAP, T]`, (a) the compact block at `h` is available and (b) a tree-state checkpoint is available at `h` itself. The `REWIND_CAP` is the librustzcash 100-block rewind cap and is not configurable from the wallet side. The wallet uses this as its scan ceiling and never scans above it.

2. **The unfiltered chain head is not exposed in the trait.** zinder still tracks `ChainEpoch::tip_height` for its own bookkeeping and other consumers (block explorers, mining dashboards) can read it via zinder's `LatestBlock` RPC. The wallet's `ChainSource` trait has one method for the wallet's view of where the chain is, and it returns the safe tip.

3. **The producer side meets the contract by writing tree-state checkpoints at a fixed stride and only advertising heights that have aged past the reorg window.** zinder writes one checkpoint every 100 blocks in both bulk-catchup and tip-follow (`zinder-ingest::chain_ingest::TREE_STATE_CHECKPOINT_STRIDE`). The `safe_tip_height` field on `ChainEpoch` is the highest height in the chain epoch whose block has aged past `reorg_window_blocks` (default 100). zinder serves the block at `safe_tip_height` via the new `LatestSafeBlock` RPC; zally's `ZinderChainSource::safe_chain_tip()` calls it directly.

4. **Contract violations surface as a typed `RequiresOperator` error.** When `Wallet::sync_inner` finds that `tree_state_at(scanned_from - 1)` did not return an anchor at exactly `scanned_from - 1`, it returns `WalletError::ChainSource(ChainSourceError::TreeStateAnchorHeightMismatch { requested, returned })` with posture `RequiresOperator`. This is non-retryable by posture (re-asking the same chain source yields the same wrong answer), surfaces clearly in the operator-facing event stream, and does not trip the wallet circuit breaker (the breaker counts retryable failures only).

5. **`realign_to_available_checkpoint` is deleted.** With the contract enforced on both sides, the realign branch is unreachable. The helper had two costs: it silently rewound the wallet by an arbitrary amount on every checkpoint mismatch (masking the producer-side bug), and it emitted a `ReorgDetected` event for a condition that was not a reorg (misleading the operator). Removing it makes the contract violation an explicit, debuggable error instead of a hidden recovery action.

6. **`safe_chain_tip` is the only vocabulary used across the contract.** zally renames every operator-facing field, event, and trait method that referenced the old `chain_tip` (which silently meant "head, may include unconfirmed") to `safe_chain_tip`. zinder renames its proto and Rust field `finalized_height` to `safe_tip_height` to avoid colliding with the consensus-level `finalized` concept Zcash NU7/Crosslink introduces. The chain-event stream family renames `Finalized` to `Safe` for the same reason.

## Consequences

- The wedge class observed on 2026-05-26 04:18 UTC is closed at the contract boundary. The wallet structurally cannot ask for an anchor in the unfinalized window because its scan ceiling never enters that window.
- The wallet sees the chain ~100 blocks behind the visible tip. For Zcash testnet at 75-second blocks the lag is ~125 minutes. The funding plane is shielded-only and already lag-tolerant; the auto-shield loop is wall-clock bucketed and not chain-anchored; the dispense path adds no extra confirmation depth on top.
- The deletion of `realign_to_available_checkpoint` makes contract violations visible. Operators see `wallet_sync_snapshot_error` with `error="...TreeStateAnchorHeightMismatch..."` and `posture=RequiresOperator`. The circuit breaker does not interact with this class.
- Operators running custom chain sources must honor the contract or accept that the wallet will refuse to sync until they do. The contract is named, documented, and observable.
- The vocabulary alignment (`safe_chain_tip` in zally, `safe_tip_height` in zinder) preserves the `finalized` namespace for the consensus-level finality concept that Zcash NU7/Crosslink will introduce.

## Alternatives considered

- **Add a parallel `safe_chain_tip()` method alongside the existing `chain_tip()`.** Rejected. The wedge class re-opens any time a future code path or custom chain source uses `chain_tip()` for the scan ceiling. Default-safe is the only choice that fights entropy.
- **Return a `SyncBounds { compact_block_ceiling, anchor_safe_ceiling }` struct.** Rejected. The two ceilings coincide by design (the stride contract makes them coincide). A struct that always returns two equal values is noise. If they diverge later, refactor then.
- **Consumer-side anchor cache in zally.** Implemented and reverted before this ADR. Cache hit rate is structurally zero in steady state because every successful sync moves `prior_height` forward by the scanned-block count, ahead of any height the cache has captured.
- **Reading commitment-tree frontiers directly from `zcash_client_sqlite` internal tables.** Rejected: tight coupling to a private schema. Every `zcash_client_sqlite` minor bump would risk breaking Zally.
- **Petitioning `zcash_client_backend` upstream to add `WalletRead::chain_state_at(h)`.** The right long-term move; deferred. The producer-side contract unblocks Zally users today without preventing the upstream improvement later.
- **Keep `realign_to_available_checkpoint` as defense in depth.** Rejected. With the contract enforced on both sides the realign branch is unreachable; keeping it means keeping a silent recovery path that masks the next contract violation. Defense in depth against an impossible failure is dead code with a comment.
- **Stride higher than 100 blocks.** Rejected: violates the contract. Any height `h` would have an at-or-below checkpoint up to `stride` blocks behind, exceeding the rewind cap.
